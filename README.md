# Консольный мессенджер (один-на-один)

Простое клиент-серверное приложение для обмена сообщениями в режиме «один-на-один».  
Сервер поддерживает одновременное подключение нескольких клиентов, сохраняет историю сообщений в текстовом файле.  
Клиент работает в консольном режиме, позволяет отправлять сообщения и просматривать историю переписки.

---

## Цель работы

Разработка клиент-серверного приложения с использованием сокетов (TCP) и текстового протокола.  
Сервер многопоточный, реализован корректный останов (graceful shutdown) по сигналу.  
Клиент имеет консольный интерфейс и устойчив к сетевым ошибкам.

---

## Требования

- Python 3.6 или выше
- Стандартная библиотека (socket, threading, signal, sys, os)

Никаких дополнительных библиотек не требуется.

---

## Установка и запуск

1. Скачайте файлы `server.py` и `client.py` в одну директорию.
2. Запустите сервер (в терминале):
   ```bash
   python server.py
   ```
   По умолчанию сервер слушает адрес `127.0.0.1` и порт `5555`.  
   Для изменения адреса/порта отредактируйте код в `__main__` или передавайте параметры через аргументы командной строки (при желании можно доработать).

3. Запустите один или несколько клиентов (в отдельных терминалах):
   ```bash
   python client.py
   ```
   Клиент запросит:
   - IP-адрес сервера (по умолчанию `127.0.0.1`)
   - Порт (по умолчанию `5555`)
   - Ваш ID (произвольная строка, идентифицирующая пользователя)
   - ID собеседника (строка, с кем вы хотите общаться)

4. После ввода данных начнётся сеанс чата.

---

## Использование клиента

В режиме чата доступны следующие действия:

- **Отправить сообщение** – просто введите текст и нажмите Enter.  
  *Примечание:* в текущей версии сообщения не должны содержать пробелов (или они будут заменены на подчёркивания). В учебных целях это ограничение допустимо; в реальном приложении протокол можно доработать.
- **Показать историю** – введите команду `/history`.
- **Выйти из чата** – введите команду `/exit` или нажмите Ctrl+C.

После отправки сообщения автоматически отображается обновлённая история, чтобы видеть отправленное сообщение и возможные новые сообщения от собеседника.

---

## Протокол взаимодействия

Сервер и клиент общаются через TCP-сокеты. Все сообщения заканчиваются символом новой строки `\n`.  
Протокол текстовый, команды и аргументы разделяются пробелами.

### Команды клиента

1. **Запрос истории**
   ```
   GET_HISTORY <user_id> <target_id>
   ```
   Пример:
   ```
   GET_HISTORY alice bob
   ```

2. **Отправка сообщения**
   ```
   SEND_MESSAGE <from_id> <to_id> <text>
   ```
   Пример:
   ```
   SEND_MESSAGE alice bob Hello_Alice!
   ```

### Ответы сервера

- **Успешный ответ на GET_HISTORY**
  ```
  OK <count>
  <from_id1>|<text1>
  <from_id2>|<text2>
  ...
  ```
  Каждая строка истории содержит отправителя и текст сообщения, разделённые символом `|`.

- **Успешный ответ на SEND_MESSAGE**
  ```
  OK
  ```

- **Ошибка**
  ```
  ERROR <описание ошибки>
  ```

---

## Хранение данных

Сервер сохраняет все сообщения в файле `messages.txt` в той же директории.  
Формат строки:
```
<from_id>|<to_id>|<text>
```
Это позволяет легко просматривать историю вручную и сохранять данные между перезапусками сервера.

## Остановка сервера

Сервер корректно завершает работу при получении сигналов `SIGINT` (Ctrl+C) или `SIGTERM`.  
При этом закрываются все сокеты, и сервер освобождает ресурсы.

```bash
^C
Получен сигнал завершения. Останавливаем сервер...
Сервер остановлен.
```

## Особенности и возможные улучшения

- **Пробелы в сообщениях**: текущая реализация заменяет пробелы на подчёркивания, чтобы не нарушать разбор команды. Для поддержки полноценных сообщений можно:
  - Передавать длину сообщения перед самим текстом.
  - Использовать экранирование.
  - Передавать текст отдельной строкой после команды.
- **Автоматическое обновление чата**: клиент не получает сообщения в реальном времени, только при ручном обновлении (`/history`) или после отправки. Можно добавить фоновый поток для периодического опроса сервера.
- **Аутентификация**: в текущей версии пользователь указывает только ID, нет паролей и регистрации. Для многопользовательской системы это можно добавить.
- **Графический интерфейс**: при желании можно заменить консольный ввод на Tkinter или другой фреймворк, но по условиям лабораторной работы требуется консоль.
